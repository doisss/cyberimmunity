## Цель работы
Целью данной работы является изучение базовых принципов кибериммунного подхода к проектированию и анализу информационных систем.  
В ходе выполнения работы рассматриваются:
- разделение системы на изолированные сущности (домены);
- контроль и ограничение взаимодействий между сущностями;
- применение и модификация политик безопасности;
- анализ поведения системы при изменении режимов работы.

---

## Описание работы
Работа выполнена в формате Jupyter Notebook и предназначена для запуска в среде **Google Colab**.  
В ноутбуке реализована упрощённая модель кибериммунной системы, состоящей из нескольких взаимодействующих сущностей, между которыми осуществляется обмен сообщениями в соответствии с заданной политикой безопасности.

Подход демонстрирует ключевую идею кибериммунитета — система считается безопасной не за счёт постоянного обнаружения атак, а за счёт архитектурных ограничений и заранее заданных правил взаимодействия.

---

## Состав системы
В демонстрационной системе используются следующие логические компоненты:
- **Сущность A** — основной активный компонент системы, инициирующий сообщения;
- **Сущность B** — вторичный компонент, принимающий и (после модификации политики) отправляющий сообщения;
- **Сущность C** — управляющая сущность, отвечающая за переключение режимов работы и изменение политики безопасности;
- **Политика безопасности** — набор правил, определяющих, какие типы сообщений и между какими сущностями разрешены.

## Задание на самостоятельную работу
1. Запустить ноутбук и изучить текущее поведение системы и логи взаимодействия сущностей.
2. Модифицировать политику безопасности таким образом, чтобы **сущность B** могла отправлять сообщения **сущности A**.
3. Реализовать циклическую отправку сообщений от сущности **A** к сущности **B** с периодичностью 5 секунд.  
   Завершение работы — по прерыванию выполнения (`CTRL + C` или остановка ячейки в Colab).
4. Настроить журналирование с использованием модуля `logging`:
   - разделить события по уровням (`INFO`, `WARNING`, `ERROR`);
   - обеспечить сохранение логов в файл.
5. Добавить **сущность C**, которая принимает только сообщения типа `change_mode` и изменяет режим работы системы:
   - режим 0 — базовая политика безопасности;
   - режим 1 — запрет всех сообщений от сущности A;
   - дополнительные режимы — по усмотрению.
6. Продемонстрировать изменение поведения системы при переключении режимов.

## Упражнения

- Измените политики безопасности так, чтобы любой текст был допустим

### Уровень "Шутки закончились"
- Измените код сущностей так, чтобы передача сообщений между WorkerA и WorkerB была двунаправленной: после получения сообщения от WorkerA WorkerB посылает ответное сообщение отправителю (например, say "done"), WorkerA должен получить сообщение и выполнить команду. С помощью plantuml.com нарисуйте диаграмму последовательности для этого сценария и вставьте в блокнот.

### Уровень "Полубог"
- Возможно, для задач этого уровня сложности имеет смысл перенести разработку кода в VS Code или PyCharm. Добавьте третью сущность (WorkerC), которая будет периодически (например, раз в минуту) запрашивать внешнюю информацию - например, курс валют с сайта ЦБ РФ (https://www.cbr-xml-daily.ru/latest.js) и отправлять WorkerA, а WorkerA в случае превышения заданного порога будет уведомлять WorkerB о событии. Очевидно, для этого сценария нужно доработать политики безопасности.
